## 1. 未来多个 hobby 的架构建议

现在只有 mixology，以后可能有 woodworking、pottery、knitting、calligraphy……如果继续走"一个 hobby 一个 html"的路子，会迅速变成下面这种地狱：

```
HobbyHub/
├── hobbydiy.html
├── mixology.html
├── woodworking.html
├── pottery.html
├── knitting.html
├── ... (20 个)
└── 每个 html 里都有重复的 nav/footer/css
```

**推荐结构**（保持纯静态也可以做到）：

```
HobbyHub/
├── index.html                ← 主页（原 hobbydiy.html 改名）
├── hobbies/
│   ├── mixology/
│   │   ├── index.html        ← 调酒列表页
│   │   └── detail.html       ← 单个配方详情（用 query 区分）
│   ├── woodworking/
│   ├── pottery/
│   └── ...
├── shared/
│   ├── css/
│   │   ├── base.css          ← 全站通用
│   │   └── components.css    ← nav/card/button
│   ├── js/
│   │   ├── nav.js            ← 通用导航
│   │   ├── gifts.js          ← 礼物系统
│   │   └── hold-reveal.js    ← 长按效果
│   └── data/
│       ├── mixology.json     ← 数据和 HTML 分离
│       └── woodworking.json
└── assets/
    └── images/
        ├── mixology/
        ├── woodworking/
        └── common/
```

## 2. 商品详情页怎么放

**两种方案，取决于规模：**

- **少量、固定**：`hobbies/mixology/detail.html?id=mojito`，一个详情页模板 + JS 读 JSON 渲染。**强烈推荐这种**，10 个配方不会变成 10 个 html。
- **每个 hobby 完全独立的详情结构**：才考虑独立文件夹。

不要每个商品一个 html 文件，会爆炸。

## 3. image 抽出来对不对？

**对，绝对要抽**。原则：
- 图片放 `assets/images/<hobby>/<item-name>.jpg`
- 命名小写连字符：`mojito-hero.jpg`、`mojito-step-1.jpg`
- 通用图（logo、icon、占位图）放 `assets/images/common/`
- 别在 HTML 里写绝对路径，用相对路径或 base URL

## 4. 要不要上 React？要不要 index？

**我的建议：先不上 React**，原因：
- 你现在是静态展示 + LocalStorage，React 的收益（组件复用、状态管理）你用不上多少
- 上 React 就要引入 Vite/构建/部署，门槛一下子拉高
- README 明确说"无需后端依赖，纯前端静态页面"，是优势别丢

**什么时候上 React**：当你要做 ① 真实购物车结算 ② 用户登录态同步 ③ 多 hobby 共享复杂筛选状态——这时候组件化才划算。届时可以直接 Vite + React，把现有 HTML 拆成组件。

**index.html 要不要**：**要**。Web 默认入口就是 `index.html`，浏览器访问 `localhost:8000/` 自动加载。`hobbydiy.html` 作为主页名字不标准，建议：

```
mv hobbydiy.html index.html
```

然后用户访问根路径就直接进主页，不用打 `/hobbydiy.html`。

## 一句话总结

改名 `hobbydiy.html` → `index.html`；新建 `hobbies/<name>/`、`shared/css/`、`shared/js/`、`assets/images/<name>/` 四个目录；详情页用一个模板 + JSON 数据驱动；React 先不急，等真要做购物车/账户再上 Vite + React。
